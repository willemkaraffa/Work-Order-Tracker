## Why this phase (read first)

The tray icon, taskbar overlay, and BrowserWindow icon all load from a
single generic file: `assets/icon.png` (942 bytes, no Gamble branding).
The in-app brand is the `GambleMark` React component (yellow "G" arc
plus blue water droplet) declared in `index.html` line 472. Users see
the Gamble mark inside the app, then a different generic icon in the
tray and on the taskbar. The request: make the tray match.

### Mechanism decision (read before editing)

Three paths considered:

1. **Ship a static PNG file** (`assets/tray-icon.png`) generated by a
   build step. Standard Electron pattern. Downside: introduces a build
   step or an external rasterizer (Sharp / Inkscape / Puppeteer), and
   the PNG drifts out of sync if `GambleMark` is ever tweaked.

2. **Runtime rasterize the React `GambleMark`** to a PNG via
   `<canvas>` in the renderer, ship the buffer to main via IPC, then
   `tray.setImage(nativeImage.createFromBuffer(buf))`. Zero new
   dependencies. Single source of truth (`GambleMark` itself). Small
   timing cost: tray shows the fallback `icon.png` for ~50 to ~300 ms
   until the renderer reports back.

3. **Inline base64 PNG constant in `main.js`**. Avoids the renderer
   roundtrip but hard-codes a pre-rendered binary in source, which
   re-introduces the drift problem (1) had. Plus the rasterization
   still has to happen somewhere.

This phase ships path **2**. It ports the MECHANISM (reuse the
existing `GambleMark` SVG, do not reinvent the vector data) per rule
2, not just the constants. The 50 to 300 ms fallback window is
covered by leaving the current `icon.png` load in place as the
initial tray image; the renderer message UPGRADES the icon when it
arrives.

If path 2 fails (e.g. the renderer crashes before the upgrade fires),
the tray stays on the fallback `icon.png`. Acceptable: that is the
current behaviour anyway.

### Out of scope (do NOT do)

- Replacing the BrowserWindow `icon:` on line 150 of `main.js`. That
  is the taskbar / Alt-Tab icon and uses the same `icon.png` file. A
  follow-up phase can swap it to a generated PNG; runtime rasterization
  does not apply (BrowserWindow icon is set BEFORE the renderer
  exists).
- Replacing the Windows overlay icon (line 230) used by
  `applyTrayBadge`. The overlay is a tiny corner badge; runtime
  rasterizing it is overkill. Leave as-is.
- macOS template-image treatment. Template images are monochrome black
  with alpha, used so macOS can tint them per menu-bar theme. The
  Gamble mark is two-color by design; making it monochrome loses the
  brand. Ship it as a regular color image. Document the trade-off in
  the risk-flags section.
- Auto-regenerating the icon when `GambleMark` changes via a file
  watcher. The runtime path already keeps them in sync on every app
  launch -- no watcher needed.

## TASK 1 -- Renderer: rasterize `GambleMark` to a PNG buffer

Add a runtime path that renders the same SVG `GambleMark` uses, into
an offscreen `<canvas>`, exports a PNG, and hands the binary to main
via IPC.

### 1a. Add a helper that draws GambleMark into a canvas

In `index.html`, find a quiet utility region (near `GambleMark`
itself is the natural home, just below its declaration -- line ~480).
Add a small helper:

```js
// Render the GambleMark vector into a PNG ArrayBuffer at a given
// pixel size. Used to push a runtime tray icon to main process.
// Drawing is via an inline SVG -> Image -> canvas pipeline; works in
// Chromium (Electron renderer) without extra deps.
async function renderGambleMarkPng(size = 32) {
  // Keep the SVG source IN SYNC with the GambleMark component above.
  // If GambleMark changes, this string must be updated too.
  const svg =
    '<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 100 100" width="' + size + '" height="' + size + '">' +
      '<path d="M 78 30 A 32 32 0 1 0 78 70 L 78 54 L 58 54" stroke="#F4C81E" stroke-width="14" fill="none" stroke-linecap="round" stroke-linejoin="round"/>' +
      '<path d="M 38 12 C 38 12 24 32 24 44 C 24 52 30 58 38 58 C 46 58 52 52 52 44 C 52 32 38 12 38 12 Z" fill="#3FA0DC"/>' +
    '</svg>';
  const blob = new Blob([svg], { type: 'image/svg+xml' });
  const url = URL.createObjectURL(blob);
  try {
    const img = new Image();
    await new Promise((resolve, reject) => {
      img.onload = resolve;
      img.onerror = reject;
      img.src = url;
    });
    const canvas = document.createElement('canvas');
    canvas.width = size;
    canvas.height = size;
    const ctx = canvas.getContext('2d');
    ctx.clearRect(0, 0, size, size);
    ctx.drawImage(img, 0, 0, size, size);
    return await new Promise((resolve) => {
      canvas.toBlob(async (b) => {
        if (!b) { resolve(null); return; }
        resolve(await b.arrayBuffer());
      }, 'image/png');
    });
  } finally {
    URL.revokeObjectURL(url);
  }
}
```

Notes:
- Two `path d="..."` strings MUST be byte-for-byte identical to the
  ones in `GambleMark`. Copy from line 475 and 476 of `index.html`
  AS THEY ARE in the file you just read; do not retype from memory.
  Verify with a follow-up grep (TASK 4 step 1).
- `Image.onload` is reliable in Electron renderer; no Chromium-version
  guard needed.
- `canvas.toBlob` is asynchronous; the outer function returns a
  Promise. Do not refactor to synchronous.

### 1b. Push the PNG to main on App mount

Find the place in `App` where Phase 15 added the
`sessionStorage` landing effect (around line 3520-3540 post-Phase-15).
Add a sibling effect AFTER that block:

```js
// Phase 16: paint the tray icon with the Gamble brand. Renderer
// rasterizes GambleMark, ships the PNG to main, main does
// tray.setImage. Runs once per app launch. If anything fails the
// tray keeps the fallback assets/icon.png from ensureTray() and
// no toast is raised -- this is cosmetic, not functional.
React.useEffect(() => {
  let cancelled = false;
  (async () => {
    if (!window.tray || !window.tray.setIcon) return;
    try {
      // 32 px is the upper bound the Windows tray will use; macOS
      // takes 22 logical px at 2x. 64 covers HiDPI on both.
      const buf32 = await renderGambleMarkPng(32);
      const buf64 = await renderGambleMarkPng(64);
      if (cancelled) return;
      if (buf32 && buf64) {
        window.tray.setIcon({ x1: buf32, x2: buf64 });
      }
    } catch (e) {
      // Swallow: cosmetic upgrade, fallback icon remains.
    }
  })();
  return () => { cancelled = true; };
}, []);
```

Notes:
- The effect's dep array is empty -- runs once on mount, not on every
  setting change. The tray icon is the brand mark; nothing in the
  brand state should re-trigger it.
- The IPC payload is an object `{ x1, x2 }` carrying two ArrayBuffers
  (the 1x and 2x bitmaps). main.js builds an Electron `nativeImage`
  representation from both -- see 2b for the rationale on shipping
  both at once.

### 1c. Bridge through `preload.js`

Read `preload.js` first. Find the `contextBridge.exposeInMainWorld`

### 2b. Add the IPC handler

After `ensureTray()` finishes its declaration block (around line 253
post-Phase-15) and before `destroyTray()` (line 255), add:

```js
ipcMain.handle('tray-set-icon', (_event, payload) => {
  if (!tray || tray.isDestroyed()) return false;
  try {
    const x1 = payload && payload.x1;
    const x2 = payload && payload.x2;
    if (!x1) return false;
    // x1 is the 1x bitmap (32x32). nativeImage.createFromBuffer
    // reads PNG/JPEG; ArrayBuffer is accepted via Buffer.from.
    const img = nativeImage.createFromBuffer(Buffer.from(x1));
    if (img.isEmpty()) return false;
    // Add HiDPI representation if present. addRepresentation needs a
    // separate nativeImage at scaleFactor 2.
    if (x2) {
      try {
        img.addRepresentation({
          scaleFactor: 2,
          buffer: Buffer.from(x2),
        });
      } catch (e) { /* HiDPI add failed -- 1x still works */ }
    }
    tray.setImage(img);
    return true;
  } catch (e) {
    return false;
  }
});
```

Notes:
- `ipcMain.handle` matches `ipcRenderer.invoke` from 1c.
- The `tray.isDestroyed()` guard mirrors the existing
  `mainWin.isDestroyed()` pattern in `applyTrayBadge` (line 217).
- `nativeImage.createFromBuffer` was introduced before Electron 4;
  this project's Electron is well past that. No version guard needed
  (verified by reading `package.json` -- if `electron` >= 12, you are
  safe).
- The handler returns `true` / `false` so a future debug session can
  inspect via DevTools.
- Do NOT call `tray.setImage(img.resize({ width: 16, height: 16 }))`.
  Electron picks the right pixel size from the `addRepresentation`
  set automatically; manual resizing here downgrades quality.

## TASK 3 -- Risk flags (mitigate or live-test before commit)

Per rule 4, every flagged risk gets either a mitigation or a live
test. Listed below with status.

1. **SVG drift between `GambleMark` and `renderGambleMarkPng`.** The
   SVG string in 1a is a copy of the JSX paths in `GambleMark`. If
   `GambleMark` is edited, the tray will not update until this
   string is too. **Mitigation:** TASK 4 grep step 1 catches drift on
   every phase-verify run. **Live test:** dismiss the launch landing
   (which uses `GambleMark` at size 88) and immediately compare the
   shape to the tray icon. They must match.

2. **macOS template-image expectation.** macOS menu-bar icons are
   conventionally monochrome black with alpha so the OS can re-tint
   them for dark/light menu-bar themes. This phase ships a two-color
   icon. **Mitigation:** documented out-of-scope above. **Live test:**
   if a macOS user later reports tray icon looks wrong on a dark menu
   bar, switch to a template variant in a follow-up phase.

3. **Fallback window during renderer startup.** `ensureTray()` runs
   on app-ready (well before the renderer mounts). The user will see
   the fallback `icon.png` for some milliseconds before the runtime
   PNG arrives. **Mitigation:** the fallback IS the current
   behaviour; this phase strictly improves on it. **Live test:** none
   needed -- if the user reports a noticeable flash, swap path 2 to a
   build-time PNG in phase 17.

4. **Renderer never reports in.** If the renderer crashes during
   startup or `renderGambleMarkPng` throws, the tray stays on
   `icon.png`. **Mitigation:** the try/catch in 1b is intentional. No
   user-visible failure. The handler in 2b also returns `false` on
   any error so future telemetry can pick it up.

5. **IPC channel name collision.** Phase 10 added `tray-action` and
   may have added other `tray-*` channels. Search `ipcMain.handle`
   and `ipcMain.on` in `main.js` BEFORE adding `tray-set-icon` --
   confirm no existing handler uses that name. (Verified during
   planning: zero hits.)

6. **`nativeImage.addRepresentation` signature.** Electron docs note
   `addRepresentation` accepts `{ scaleFactor, width, height,
   buffer, dataURL }`. We pass `scaleFactor` and `buffer`; Electron
   infers width/height from the PNG header. If Electron version is
   older than ~10 the call may behave differently. **Mitigation:**
   the outer try/catch isolates the 2x failure from the 1x success
   path. 1x always works.
