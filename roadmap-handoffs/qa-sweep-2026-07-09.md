## Q1 — Persistent text lock-out bug  [INSTRUMENTED + hardened; NOT claimed fixed]
Symptom (memory bug_note_card_input_lock): after editing a SAVED note, all text entry
"freezes until restart".
Static findings: `NoteCard` (detail.jsx:719) is clean — real `<textarea>`, focus set via
effect; while editing, `activeElement` is the textarea so the global type-to-search handler
passes through. The suspect is the GLOBAL keydown in `useTypeToSearch` (search-hook.js): it
`preventDefault()`s every printable key whenever `activeElement` is not an input, redirecting
it into the module search box. After a note save the textarea unmounts and focus falls to
`<body>`, so subsequent keys are eaten/redirected — a plausible "typing does nothing here"
lock. Memory rule: DON'T blind-patch; capture `activeElement` on the next repro.
Actions taken (both safe, neither is a guess at NoteCard):
  (a) HARDEN `useTypeToSearch`: bail when there is no mounted search input to receive the
      key (`!inputRef.current`) instead of `preventDefault`-eating it into the void.
  (b) INSTRUMENT: `window.__lockDebug()` (defined in search-hook.js) prints
      `activeElement` tag/id, isTyping, and `__modalOpen`. When the lock recurs: open
      devtools console, run `__lockDebug()`, paste the result -> that pins the mechanism.
Status: hardened + instrumented. Needs one live repro + `__lockDebug()` output to close.
